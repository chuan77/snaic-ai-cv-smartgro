# Project Handoff — 2026-07-09

## What This Project Does

SmartCart AI Checkout Assistant: a 5-stage pipeline (Days 1-5) that turns unannotated grocery classification photos into a working YOLO11 object detector by programmatically compositing product crops onto synthetic backgrounds, then serves live detections through a checkout UI that prices items from a catalog. Two serving surfaces exist: a Gradio dashboard (`main_day5_deploy.py`, Python-only) and a React/TypeScript/Vite frontend (`frontend/`) backed by a FastAPI service (`main_api_server.py`). All ML code targets Apple Silicon (`torch.device("mps" if torch.backends.mps.is_available() else "cpu")`). See `CLAUDE.md`, `docs/handoff-2026-07-07.md`, and `docs/handoff-2026-07-08.md` for prior context.

## What Was Completed Today

### 1. Fine-grained product variant resolution (this session)

**Problem:** the cart showed a generic label for two categories — "Chocolate Bar" / `Snacks/Chocolate-Bar · 74%` — instead of the actual product (e.g. Cadbury RoastAlmond). Root cause: YOLO only detects 5 coarse classes, and `Snacks/Chocolate-Bar`/`Ready-To-Eat/Instant-Noodles` had no brand/variant breakdown in the catalog, even though the raw Label Studio annotations (`dataset/raw_photos/classes.txt`) *did* know the specific variant per photo — it was being discarded on import.

**Why not just retrain YOLO on the fine-grained classes:** the project's existing 83-class retrain (`runs/detect/retrain_20260708_184220*`, ~0.957 mAP50) proves fine-grained retraining works *when there's enough data* (45–65 photos/class for Fruit/Packages/Vegetables). The raw chocolate/noodle photos only yield ~2–4 photos per brand/variant — far too sparse to train a reliable detector class. Confirmed with the user before proceeding.

**Solution implemented:** keep YOLO detecting the 5 coarse classes; add a second stage that crops the detected box and matches it via DINOv2 nearest-neighbor against the gallery, scoped to that category's variants (frozen pretrained features work fine even with a handful of reference photos per variant).

- `src/data/annotation_import.py` — `import_label_studio_export()` now nests crops by variant subpath instead of flattening into one coarse folder. `classify_category()` returns `(keyword, dest_dir)`, picking the **longest** matching keyword (fixes a real collision: `Myojo/Chicken` is a substring of `Myojo/ChickenAbalone` once fine-grained folders exist). New `variant_subpath()` preserves Brand/Variant nesting when present (the classes.txt scheme the user confirmed as authoritative going forward — see the file's own history for the two competing naming schemes).
- Re-ran `import_annotations.py` + `main_day1_catalog.py` against the existing local `dataset/raw_photos` — `artifacts/catalog_prices.csv` and the DINOv2 gallery now have 18 chocolate-bar + 21 instant-noodle priced variants (e.g. `Ready-To-Eat/Instant-Noodles/Nissin/328-KATONG-LAKSA`).
- `src/models/variant_resolver.py` (new) — `VariantResolver.resolve(coarse_class_name, crop)` does cosine-similarity nearest-neighbor over gallery rows nested under that coarse prefix; falls back to the coarse label untouched when nothing finer exists (safe to call unconditionally). Reuses `src/models/dino_extractor.py`'s `DinoFeatureExtractor` (previously unwired).
- `src/deploy/detector.py` / `src/deploy/api_server.py` — new `get_variant_resolver()` singleton (env-configurable via `SMARTCART_GALLERY_INDEX_PATH`/`SMARTCART_GALLERY_META_PATH`); `predict_endpoint` now runs `resolve_variants()` on every detection before returning.
- `frontend/src/components/ProductCard.tsx` — subtitle now wraps to 2 lines at a smaller font (`line-clamp-2 whitespace-normal break-words text-[10px] leading-tight`) instead of truncating, since fine-grained SKUs run much longer.
- Verified end-to-end via a headless Playwright session against a real synthetic checkout scene: cart correctly showed "Ramen Hokkaido Miso" / `Ready-To-Eat/Instant-Noodles/Nissin/Ramen-Hokkaido-Miso · 57%`, wrapped, correctly priced.

### 2. Active-learning capture → Label Studio push (this session)

Discovered the active-learning capture pipeline (`src/deploy/active_learning_capture.py`) had been running against `/predict` with nothing ever consuming it — 153 unreviewed captures sat in `artifacts/active_learning_staging/` (118 zero-detection, 35 low-confidence including 10 chocolate-bar + 10 instant-noodle). Pushed the 35 low-confidence ones to Label Studio project 1 for human review.

- `src/deploy/label_studio_push.py` — `push_staging_dir()` gained a `min_detections` parameter (TDD); new `get_push_min_detections()` reads `SMARTCART_PUSH_MIN_DETECTIONS` (added to `.env.example`, default 0).
- **Two real gaps found while actually running this against the live Label Studio instance** (see `docs/superpowers`-adjacent memory for detail, not yet fixed in code):
  1. `.env` is only loaded inside `src/deploy/api_server.py` — standalone scripts (`push_captures_to_label_studio.py`, `retrain_from_label_studio.py`) never see it. Worked around with `set -a; source .env; set +a` for this run.
  2. This Label Studio instance issues JWT-style Personal Access Tokens for `LABEL_STUDIO_API_KEY`, not simple legacy keys — `push_tasks()`/`fetch_task_image()`/`export_project()` all send `Authorization: Token …`, which 401s. The real fix is `POST /api/token/refresh` → `Authorization: Bearer <access>`. Worked around with a one-off script bypassing `push_tasks()`; the three call sites in `label_studio_push.py`, `label_studio_backend.py`, and `label_studio_export_pull.py` still need this fix before push/pull/the LS ML-backend callback will work again unassisted.
- **Self-caught mistake:** a rewritten push command dropped `SMARTCART_PUSH_MIN_DETECTIONS=1`, pushing all 153 captures instead of 35. Caught it, confirmed the exact 118 over-pushed tasks by matching image URLs, deleted them from Label Studio via its API, and restored their local files — final state matches what was intended (project 1: 114 pre-existing + 35 new = 149 tasks; local staging: 118 unpushed + 35 in `pushed/`).

## Running the Stack

```bash
# Install/sync Python deps
uv sync

# Configure (first time only)
cp .env.example .env               # then add LABEL_STUDIO_API_KEY, SMARTCART_LS_PROJECT_ID
cd frontend && cp .env.example .env && cd ..

# Run the backend (.env only auto-loads for the API server itself — see gap #1 above)
uv run python main_api_server.py     # http://localhost:8888 per this repo's .env

# Run the frontend (separate terminal)
cd frontend && npm install && npm run dev   # http://localhost:5280 per this repo's .env

# Run tests
uv run pytest -v
cd frontend && npm run build && npm run lint

# Push staged active-learning captures to Label Studio (needs `.env` sourced manually — see gap #1)
set -a; source .env; set +a
SMARTCART_PUSH_MIN_DETECTIONS=1 uv run python push_captures_to_label_studio.py
```
